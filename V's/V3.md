# FlightGear AI ATC — V3: Groundnet Taxiway Naming & Feature Backlog

**Date:** 2026-06-24

## Overview

V3 captures two things: (1) a critical data-format finding about groundnet taxiway naming that clarifies the feasibility of named-taxiway routing, and (2) a comprehensive backlog of post-V1 feature ideas organized by impact and effort.

---

## 1. Groundnet Taxiway Naming Finding

### The Forum Claim

A FlightGear forum reviewer asserted that groundnet XML files don't contain taxiway letters, so the add-on could only degrade to "taxi to runway XX" without intermediate taxiway names.

### Verification Against KSFO Fixture

We verified this claim against the committed `fixtures/KSFO.groundnet.xml` and found **the claim is partially true but misleading**:

**Schema:** The `<arc>` (taxiway segment) element *does* have a `name` attribute.
```xml
<arc begin="1234" end="5678" name="A" isPushBackRoute="0"/>
```

**Data Distribution in KSFO:**
- **Total arcs:** 3,131
- **Named arcs:** 150 total
  - Real taxiway letters (A–G): **107 (3.4%)**
  - Generic placeholders ("Route", "Startup Location"): **43 (1.4%)**
- **Empty/unnamed arcs:** 2,981 (95.2%)

**Spatial Pattern:** The named arcs **cluster at runway entries and major intersections** — exactly where ATC would invoke taxiway names. This is the sweet spot for phraseology clarity.

### Parser Graceful Degradation

The repo's design **already handles sparse naming** with three tiers:

1. **Tier 1 (Online):** Use real names where present; `parser_ai.py` uses a Gemini prompt to infer/label empty segments from groundnet geometry when online.
2. **Tier 2 (Offline, named):** Fall back to inferred names from cached picture if names were labeled on a prior online session.
3. **Tier 3 (Offline, sparse):** If the path is only sparsely named, `phraseology.py` conditionally omits the `via` clause, degrading to "taxi to runway 28L" alone.

### Verdict & Refinement

**Keep named-taxiway routing; do not abandon it.** The design is sound.

**Refinement to adopt:** **Sparse-name suppression** — suppress the `via` clause when the traced route contains too few named segments. For example, "taxi via Bravo" with no surrounding context is more confusing than "taxi to runway 28L" alone.

- **Gate the clause on a coverage threshold:** e.g., ≥ 3 named segments in the path, OR names covering > 30% of path length (by arc count).
- **Rationale:** Low coverage triggers more pilot confusion than zero names; the threshold prevents one-off names from polluting the instruction.
- **Implementation:** In `phraseology.py`, measure named/total arc ratio on the routed path and only include `via` if threshold met.

## 2. Feature Backlog

### Top Priorities (Recommended Next Work)

These five features offer the highest impact for near-term effort and should be sequenced after V1 stabilizes.

#### 1. METAR-Driven Runway Selection (High Impact / Low Effort)

**What it does:** Wind + per-aircraft crosswind limit + runway heading determine the active runway; stops ATC assigning a runway into a strong tailwind.

**Why:** Current V1 assigns runways deterministically. In real operations, wind drives runway selection. A single aircraft can reject an assigned runway if crosswind exceeds certified limits.

**Integration:** Fetch METAR from `aviationweather.gov` API; extract wind direction/speed. Compute crosswind component for each runway. Compare to aircraft type (read `/sim/aircraft-id` from property); look up certified crosswind limit from a simple CSV or Gemini (aircraft + crosswind → limit). Filter eligible runways; prefer the longest to minimize taxi distance if both are viable.

**Note:** A "trace path to takeoff runway" implementation building on this is already underway.

---

### Implementation Status — Parking Spot → Takeoff Runway (HALF COMPLETE)

**Status:** Core routing logic and phraseology integration are working. Runway data availability and in-sim end-to-end testing remain incomplete.

#### DONE (Working on Tree, Not Yet Committed)

- **New module `sidecar/runway_selection.py`** with the following functions:
  - `headwind_component(runway_heading, wind_dir, wind_kt)` — Scores a runway against wind; computes crosswind and headwind components.
  - `select_departure_runway(picture, *, wind_dir=None, wind_kt=None)` — Selects the active departure runway by best headwind and least crosswind. Wind is an input parameter (pure/testable). Graceful fallback: longer runway when wind is absent.
  - `start_node_for_position(picture, lat, lon, *, parking_id=None)` — Resolves aircraft's gate/parking or lat–lon to a start graph node. Prefers parking ID (which is already a valid graph node via its pushback arc).
  - `on_runway_node_for_position(picture, lat, lon)` — Helper to find nearest on-runway node.
  - `runway_entry_node(picture, runway)` — Finds the hold-short/entry node for the selected runway.
  - `taxi_to_runway(picture, callsign, lat, lon, *, wind_dir=None, wind_kt=None, parking_id=None)` — End-to-end glue: selects runway → resolves start node → resolves runway-entry node → invokes existing A* (`route_taxiways`) → generates clearance via existing phraseology. Reuses existing routing and phraseology; does not duplicate A*.
- **Graceful degradation preserved:** When the routed path has no named taxiway segments, clearance degrades to "taxi to runway XX" with no `via` clause (existing contract honored).
- **Test coverage:** `tests/test_runway_selection.py` with 17 new tests; all passing. Full test suite: 105 passed, 0 failed.

#### REMAINING TO COMPLETE

1. **Runway data not available from groundnet XML alone.** `Runway` objects come from FlightGear's in-sim `airportinfo` at runtime, not from `groundnet.xml`. Parsing `fixtures/KSFO.groundnet.xml` alone yields `picture.runways == []`. Consequence: `select_departure_runway` and `runway_entry_node` are unit-tested only with synthetic runway inputs — the wind-based selection path is **not exercised end-to-end** against the committed fixture or in-sim.

2. **In-sim verification needed.** Must run inside FlightGear (or construct a runway-populated `AirportPicture`) to confirm runway selection + entry-node resolution work with real runway data. Includes validating that `runway_entry_node` correctly maps the selected runway end to a real hold-short/entry graph node.

3. **Main-flow integration.** Currently runway is supplied by Nasal (`REQ_RUNWAY`). To auto-select the departure runway in the sidecar, must wire `select_departure_runway` / `taxi_to_runway` into the main event loop to choose runway instead of requiring Nasal to pass one.

4. **Live wind/METAR source not wired.** `wind_dir` / `wind_kt` must currently be passed in by the caller. Completing the feature means fetching live wind from METAR (aviationweather.gov) and feeding it into `select_departure_runway`. This connects to the "METAR-driven runway selection" top-priority item above.

5. **Sparse-name suppression refinement** (documented elsewhere in this document) remains documented-only and not yet implemented in code. The parking→runway clearance output would benefit from gating the `via` clause on named-segment coverage threshold (e.g., ≥ 3 named segments or > 30% of path).

#### 2. Callsign + Aircraft-Type Personalization (High Impact / Low Effort)

**What it does:** Read `/sim/multiplay/callsign` and `/sim/aircraft-id`, inject into Gemini prompts so ATC calls are personalized (e.g., "N12345, taxi to runway 28L") and aircraft-type appropriate.

**Why:** Immersion. Generic "aircraft" → "Citation X" is a big difference (a 7-seater vs. a heavy). Payware ATC tools do this; it should be table stakes.

**Integration:** Add callsign + aircraft type to `phraseology.py` Gemini system prompt. Add type-specific crosswind/runway assignment filters in runway selection.

---

#### 3. Replay/Regression Testing from Session Captures (High Impact / Medium Effort)

**What it does:** Record property snapshots + ATC exchanges to JSONL per session; add a `--replay` mode that runs routing/phraseology against golden files with no FlightGear running.

**Why:** Regression testing. Once you ship a phrase or route that works, you want to know the next refactor didn't break it. Golden-file replay is the pattern.

**Integration:** Extend `main.py` event loop to log all state snapshots + requests/responses to a session JSONL file. Add a `--replay <session.jsonl>` mode that reads the file, replays requests, and compares output against golden responses (or just logs diffs for manual inspection).

---

#### 4. Phraseology Grading + Debrief (High Impact / Medium Effort)

**What it does:** Log pilot selections vs. correct ICAO phrasing; generate a scored end-of-session debrief (e.g., "You readback 'roger' instead of 'wilco'" or "You violated hold-short by 50ft").

**Why:** **No payware ATC tool does pilot grading.** This is a real differentiator for flight training. Turns the add-on from a tower/controller sim into a training tool.

**Integration:** Extend `phraseology.py` to score pilot readback against expected ICAO format (use Gemini with structured output for rubric scoring). Store grade + feedback in session log. At end of session, synthesize a debrief dialog showing score, errors, and tips. Optional: export as a learner logbook entry (PDF, text, or JSON).

---

#### 5. SID Assignment from Filed Route (High Impact / Medium Effort)

**What it does:** Read `/autopilot/route-manager/`, derive the departure procedure (FAA CIFP data or Gemini+Search), validate against active runway.

**Why:** IFR completeness. Currently, the add-on assigns a runway but not a SID. Real IFR clearance always includes a SID (or "no SID" if the airport has none). This closes the gap.

**Integration:** Parse route manager waypoints → first waypoint (departure fix). Use Gemini + Google Search API to fetch the runway-specific SID from FAA CIFP (US) or derive it (non-US). Validate the SID against active runway. Inject into clearance-delivery phraseology: "Cleared as filed via [SID], climb 2000."

### Other Ideas (Medium/Low Priority)

Include these in the backlog with brief rationales, grouped by theme:

#### Data & Environment
- **NOTAM injection into the taxi graph** — remove closed segments and relay NOTAMs in context (e.g., "taxiway A is closed, taxi via B instead"). Requires NOTAM parser + sidecar integration.
- **SIGMET/AIRMET weather-deviation advisory relay** — fetch SIGMETs; if significant weather intersects route, offer deviation advisory. Requires weather API + geospatial logic.
- **FAA CIFP procedure database** — authoritative SID/STAR/approach text; Gemini fallback for non-US. Reduces reliance on Gemini inference.
- **D-ATIS real-world active runway** — fetch D-ATIS, extract active runway to validate against computed runway. Adds real-world ground truth.

#### Taxi & Movement
- **Conditional/progressive taxi monitoring** — auto-issue next segment at hold-short points via position polling. Extends the single upfront taxi clearance into a reactive, progressive sequence.
- **Missed-approach/go-around detection** — altitude+VS+groundspeed signature detection; re-sequence traffic if pilot goes around. Requires live altitude/VS monitoring.
- **Graceful reconnect + clearance-state persistence** — SQLite session state across FG restarts; resume clearance state without re-filing.

#### UX & Safety
- **ATIS information-code cycling** — persist ATIS info code (Alpha, Bravo, …) per airport; advance only on material METAR change. Realistic ATIS ops.
- **Readback training mode** — inject a deliberate error (e.g., wrong runway) in readback; check whether the pilot catches it. A teaching feature.
- **Sidecar health dashboard** — localhost FastAPI page: FG connection status, cache hit/miss rate, Gemini latency, active TTS voice, session log. Observability.
- **Accessibility: high-contrast/large-font dialog + plain-text transcript file; keyboard-only ATC shortcuts.** Opens to a broader user base.
- **Logbook integration** — write machine-readable `logbook.json` per session (callsign, airport, runway, SID, duration, grade). Cumulative training log.

#### Tower & Multi-Aircraft
- **Tower mode: multi-aircraft sequencing** — use `/ai/models/aircraft[n]` to build a traffic picture; sequence multiple AI aircraft for landing/departure. Addresses the biggest Red Griffin gap.
- **Shared-cockpit ATC radio assignment** — captain/FO active-seat gating; only the active pilot can request a clearance. Multiplayer realism.

## 4. Staging Recommendation

**Phase 1 (Current):** Complete V1 (taxi routing + phraseology + telnet/Nasal bridge + TTS).

**Phase 2 (Recommended Next):** Sequence the five top-priority ideas in order:
1. METAR-driven runway selection
2. Callsign + aircraft-type personalization
3. Replay/regression testing
4. Phraseology grading + debrief
5. SID assignment

Each is a self-contained feature that doesn't block the others. Completing these five raises the add-on from a proof-of-concept to a credible alternative to payware ATC tools.

**Phase 3+:** Backlog items and tower mode as engineering capacity allows.

## 5. Open Questions

- **CIFP data source:** Should we license FAA CIFP (annual subscription) or rely on Gemini + Search API to infer SIDs? Gemini is fast and free but less authoritative; CIFP is authoritative but costs money.
- **Crosswind CSV:** Where do we source aircraft certified crosswind limits? A curated CSV, or Gemini inference from type name?
- **Grading rubric:** What ICAO readback errors should trigger deductions? (Wrong altitude? Wrong runway? Wrong phonetics?) Define a simple rubric.
