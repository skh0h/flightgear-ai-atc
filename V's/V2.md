
# FlightGear AI ATC — V2: Expanded Vision & Feature Roadmap

**Date:** 2026-06-24

## Overview

**V1** (the build-it-working plan in `PLANS/implementation-plan.md`) focuses on delivering a complete end-to-end taxi-clearance demo: config → airport pipeline → routing → telnet/Nasal bridge → in-sim TTS. V2 captures the broader product vision and cross-platform / TTS / transport decisions confirmed in this session. V2 is **proposed and not yet sequenced into the build plan** — V1 remains the active milestone.

---

## 1. Confirmed Decisions

### Milestone Strategy
Finish the documented V1 first — complete Phases 2→3→4 (airport picture pipeline → A* routing + phraseology → telnet/Nasal bridge + TTS) to a working in-sim taxi-clearance demo. **Do NOT pull voice-input/STT or AI-traffic sequencing into V1.** Tier-1/Tier-2 features layer on top after V1 is complete and shipped.

### Cross-Platform Support
Target **Mac AND Windows**. Many users run Windows; the Python sidecar is already cross-platform (Python + FlightGear HTTP/telnet behave identically on both). The Mac-specific TTS piece (`say`) is now resolved via a pluggable TTS backend.

**Action:** Ship launch scripts for both OSes (`.command`/shell for Mac, `.bat`/PowerShell for Windows); no installer yet.

### TTS / Voice UX Strategy

**Text Display (Priority 1):** ATC reply must be shown as **TEXT prominently at the TOP** of the dialog, plus a voice picker offering ~10–20 selectable voices.

**Engine Strategy:** All TTS engines behind a planned `tts.py` swappable backend (ABC interface):

| Engine | Why | Trade-offs |
|--------|-----|-----------|
| **Piper** (PRIMARY) | Free, offline, neural quality, first-class Windows + Mac support | Best practicality default |
| **Kokoro** | Higher quality than Piper, better naturalness | Heavier model, wants GPU for responsiveness |
| **Cloud** (ElevenLabs / Google / Azure) | Highest quality | Requires API key, per-use cost, network latency |
| **OS-native** | Zero-setup fallback | `say` on Mac, SAPI5 on Windows (robotic) |
| ~~Festival / eSpeak / Flite~~ | Rejected | Pre-neural, robotic, poor Windows support |

**One-liner:** Piper = best default; Kokoro = best free upgrade; ElevenLabs = best if paying.

### Read Transport (Sidecar ↔ FlightGear)

Use the **FlightGear HTTP JSON API** (`--httpd`) for ~10 Hz state reads (property polling). Telnet is ~5 commands/sec — too slow for high-frequency updates.

Use **telnet / Nasal only for writes** (clearances, frequency changes).

**Enhancement Option:** WebSocket `/PropertyListener` for push-style property-change events (instead of polling). Deferred to Phase 2+ if latency becomes a blocker.

### Architecture Guardrail: Python-Authoritative Clearance State

Keep the **clearance state machine in Python**. Gemini generates language and phrasing only; it is **never** the authority on what was actually cleared (altitude, runway, squawk, controller position).

**Why:** An arxiv study found LLM ATC agents routinely violate their own separation rules when they hold the authoritative state. The LLM stays out of safety-critical decisions. Closest prior art: `fgatc` (bartacruz, GitHub) — a Python sidecar with Dijkstra groundnet routing.

## 2. Feature Roadmap

### Tier 1 — Highest Impact (Enabled by Current Integration Surface)

| Feature | What it does | Key integration point |
|---------|-----------|----------------------|
| **IFR Clearance Delivery** | Pilot calls for clearance; proper C-D-A-F-S phraseology (clearance limit, departure, altitude, frequency, squawk); readback required. *The single feature that makes it feel like real IFR.* | Detect Delivery/Ground frequency via COM1 poll; read aircraft type from `/sim/aircraft-id` |
| **Taxi Instructions with Groundnet Routing** | Parse groundnet XML, build node graph, shortest path (parking → runway hold-short), progressive taxi string. Already Phase 2–3 plan; KSFO fixture in place. | Groundnet XML loader; A* pathfinding; real-time aircraft position |
| **ATIS from Live METAR** | Fetch METAR (aviationweather.gov), Gemini generates phonetic ATIS with info code (Alpha, Bravo…), broadcast on ATIS frequency. Cheap, high realism. | HTTP fetch + Gemini; ATIS frequency polling from airport freq data |

### Tier 2 — Strong Immersion (Moderate Effort)

| Feature | What it does | Integration complexity |
|---------|-----------|----------------------|
| **Voice I/O Loop** | Faster-Whisper (STT) + Silero VAD for push-to-talk detection + Piper/Kokoro (TTS); optionally inject audio through FGCom-mumble on tuned frequency. *Biggest single immersion driver.* | Deferred out of V1; requires audio capture + Mumble integration |
| **Departure / Approach Handoffs** | Detect climb-through altitude or top-of-descent; issue "contact Departure/Approach on …"; confirm via detecting new tuned frequency. | Read `/sim/autopilot/settings/target-altitude`; compare to current; trigger at threshold |
| **Arrival Sequencing & Vectoring** | ~30nm out, issue vectors/speed/intercept. Reads `/ai/models/aircraft[n]` to build basic traffic picture. Addresses sequencing gap. | AI/MP property space; distance/bearing calc; simple priority queue |
| **Non-Standard Request Handling** | Pure LLM value: "unable assigned altitude, request FL280," weather deviations, etc. | LLM flexibility + Python approval layer; no new integration |

### Tier 3 — Later (Nice-to-Have)

- **Controller Personality Profiles** — terse Center vs. chatty tower via Gemini system prompt.
- **Emergency Handling** — MAYDAY / PAN-PAN with priority handling and vectors to nearest airport.
- **Readback-Error Correction** — detect wrong altitude/runway in pilot readback, issue correction.

## 3. Second-Layer Ideas / Backlog

### Immersion & UX

- **Route ATC audio through FGCom-mumble** — inject synthesized voice as audio on the tuned frequency so it comes through the in-sim radio spatially, not detached system audio.
- **Radio-quality effects** — static/squelch/clipping on TTS output; pairs with controller personalities.
- **Frequency tuning IS the trigger** — poll COM1, map MHz → controller position (GND/TWR/APP…) from airport freq data to select controller + persona.
- **Progressive taxi depth** — explicit hold-short and runway-crossing clearances by comparing live position to groundnet nodes (not one upfront string).
- **VFR advisory mode** — traffic advisories, pattern/circuit work as a second mode beyond IFR — a payware differentiator.

### Technical & Data Infrastructure

- **WebSocket `/PropertyListener`** — push property-change events (vs. polling) for ~1ms responsiveness.
- **Data-pipe gap to design early:** Frequencies, runway thresholds, and parking/gate positions must reach the sidecar via Nasal `airportinfo()` + `/position` props. Phase 2–3 routing depends on it.

## 4. Guardrails & Lessons Learned

### From Prior Art & Research

- **Keep the LLM out of safety-critical decisions.** An arxiv study found LLM ATC agents violate their own separation rules when they hold state. Hence: Python-authoritative clearance state machine.
- **Design around SayIntentions.AI's pain points:**
  - STT mis-hears → implement readback confirmation loop.
  - Readback errors go uncaught → explicit validation before applying clearance.
  - Server latency → local fallback / offline Piper as default.
  - Implausible runway selection → filter by wind direction + runway status from METAR + active runways.

## Appendix: Roadmap Rationale

### Why Finish V1 First?

1. **Completeness:** A taxi-clearance demo end-to-end (from aircraft startup → parking clearance received) proves all integration layers work.
2. **Credibility:** Ships a working feature set before adding complexity.
3. **Foundation:** V1 establishes data-pipe, TTS, and state-machine patterns that V2 features reuse.

### Why These Tier-1 Features?

- **IFR Clearance Delivery** is the highest-fidelity ATC interaction and the one most payware-ATC users expect.
- **Taxi + Routing** is the core Phase 2–3 commitment; it's a self-contained, shippable feature.
- **ATIS** is nearly free (just fetch + Gemini + frequency broadcast) and immediately raises immersion.

### Why Python-Authoritative State?

Gives the LLM a clear role (language generation) and removes it from safety-critical logic. Mirrors the BeyondATC pattern: AI generates *language*, Python enforces *rules*. This separation also makes the system auditable — a human can review what was actually cleared vs. what Gemini said.
